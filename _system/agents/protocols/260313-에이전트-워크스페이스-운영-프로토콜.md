# 에이전트 워크스페이스 운영 프로토콜

## 1. 공통 절차

1. 작업 대상 파일의 성격을 판별한다.
2. 파일명과 코드가 필요하면 `_system/agents/indexes/260313-파일명-코드-참조규칙.md`를 먼저 확인한다.
3. 라우팅 인덱스에서 목적지 폴더를 찾는다.
4. 기존 동일 파일 존재 여부와 상태(`draft`, `live`, `final`, `arch`)를 확인한다.
5. 이동/생성/수정 작업을 수행한다.
6. 결과를 로그 또는 메모리 문서에 남긴다.

## 2. 파일 흡수 프로토콜

### 문서
- `Markdown` 원본은 우선 보존한다.
- `PDF`, `DOCX`는 같은 주제 문서권역 안에 보관한다.
- 상태가 불명확하면 `draft`로 두고 확정 문서를 덮지 않는다.
- 새 문서를 만들 때는 `YYMMDD-제목.ext` 패턴을 적용한다. (상세: `_system/rules.md`)

### 스프레드시트와 CSV
- 재사용 가능한 것은 `01_ops/templates/` 또는 `04_studio/templates/`
- 실제 운영본은 `01_ops/` 또는 해당 실행 폴더
- seed 데이터는 `01_ops/templates/notion-seeds/`

### 자산
- 운영 문서에 삽입되는 이미지라도 원본은 `04_studio/assets/`
- 단순 참고용 브랜딩 사례는 `05_library/research/references/brand/`

## 3. 충돌 해결 순서

1. 이름과 해시가 모두 같으면 skip
2. 이름은 같고 내용이 다르면 preserve both
3. `live/final` 문서를 `draft/arch`보다 우선
4. 오래된 테스트 산출물은 `05_library/seasons/2025-pre/legacy-hold/archive/05_Snapshots/`

## 4. 메모리 운영

- 현재 판단 중인 것: `_system/agents/memory/active-context/`
- 구조 관련 합의: `_system/agents/memory/decisions/`
- 변경 이력: `_system/agents/memory/change-log/`

## 5. 수정 원칙

- 큰 구조 개편 전에는 반드시 인덱스 문서를 먼저 업데이트한다.
- 루트 정리는 보수적으로 한다.
- 숨김 파일과 시스템 파일은 운영 자산으로 취급하지 않는다.
- 파일명 코드 체계가 바뀌면 관련 `README.md`, `AGENTS.md`, 인덱스, 코드 참조 문서를 함께 갱신한다.
