# [클럽명] OS — 에이전트 아키텍처

## 0. 설계 철학

[클럽명]는 콘텐츠 공장이 아니라 **살아있는 지식 공동체**다.
따라서 [클럽명] OS의 에이전트들은 생산 파이프라인의 부품이 아니라,
공동체가 **생각하고, 기록하고, 스스로를 돌아보고, 성장하는** 과정을 돕는 존재다.

### 핵심 원칙

1. **축적 우선** — 모든 에이전트 작업은 [볼트명] 볼트에 마크다운으로 남는다. 휘발되는 것은 없다.
2. **Source → Publish는 방향 중 하나일 뿐** — 발행은 공동체 활동의 결과물 중 하나지, 유일한 목적이 아니다.
3. **공동체는 자기 자신을 연구한다** — 내용(PKM, GTD, AI)을 탐구하는 것과 운영 구조를 디벨롭하는 것은 동등하게 중요하다.
4. **기록은 관리와 다르다** — 파일을 정리하는 것과 무슨 일이 있었는지 기억하는 것은 별개의 행위다.

## 2. 각 에이전트 상세

### 2.1 [클럽명]-ops (조율자)

> "지금 공동체에 무엇이 필요한가?"

오케스트레이터. 전체 시스템의 맥박을 읽고 다른 에이전트에게 일을 요청한다.
태스크 매니저가 아니라 **감각 기관**에 가깝다.

- **주간 리뷰**: 볼트 전체를 읽고 현황 보고 + 추천 작업 목록 생성
- **분기 회고**: 지난 분기의 세션, 멤버, 성장, 문제를 종합 분석
- **작업 조율**: "이번 주에 chronicle이 할 일, thinker가 할 일"을 판단
- **건강 체크**: draft 문서 체류 기간, 발행 대기열 적체, 빈 폴더 현황

읽기: 전체 / 쓰기: `01_ops/strategy/`

---

### 2.2 [클럽명]-chronicle (사관)

> "무슨 일이 있었는가?"

클럽의 **기억**을 담당한다. 세션이 끝나면 기록하고, 결정이 내려지면 남기고,
의미 있는 대화가 오가면 요약한다. 이 에이전트가 없으면 클럽의 과거는 사라진다.

- **세션 기록**: 세션 후 진행 내용·핵심 인사이트·참여자 반응 정리
- **결정 로그**: 운영 회의에서 내린 결정과 그 맥락을 기록
- **마일스톤**: 클럽의 중요한 순간(첫 세션, 첫 멤버 합류, 첫 쇼케이스 등) 기록
- **대화 회수**: Discord에서 의미 있는 논의가 발생했을 때 요약·보존

읽기: 전체 / 쓰기: `02_sessions/`, `01_ops/strategy/`, `01_ops/members/`

---

### 2.3 [클럽명]-thinker (탐구자)

> "이 주제를 더 깊이 파면 무엇이 나오는가?"

클럽이 다루는 **주제 영역**을 깊이 탐구한다.
PKM, GTD, Deep Work, AI 자동화, 시각적 사고, 심리학적 학습법 등.

- **연구 노트 작성**: 특정 주제에 대한 깊은 탐구 결과를 정리
- **도구 리뷰**: 새로운 도구·방법론·워크플로우를 실험하고 평가
- **세션 씨앗**: 연구 결과를 바탕으로 세션 주제 제안
- **지식 지형도**: 클럽이 다룬 주제의 관계와 빈 영역을 시각화

읽기: `05_library/research/`, `02_sessions/backlog/`, 웹 / 쓰기: `05_library/research/`

---

### 2.4 [클럽명]-meta (메타 설계자)

> "우리 시스템은 잘 작동하고 있는가?"

클럽 **자체를 연구 대상으로 삼는** 에이전트.
내용이 아니라 **구조와 과정**을 탐구한다.

- **운영 R&D**: 세션 포맷, 멤버 온보딩 흐름, 역할 구조, 도구 조합을 실험·개선
- **시스템 진화**: [클럽명] OS 자체의 개선안 제안 (새 템플릿, 새 워크플로우, 에이전트 역할 조정)
- **커뮤니티 패턴 분석**: 무엇이 멤버를 끌어오는가, 무엇이 에너지를 떨어뜨리는가
- **비교 연구**: 다른 학습 공동체·동아리의 운영 사례를 조사하고 적용점 탐색
- **회고 합성**: 여러 세션 회고에서 반복되는 패턴 추출

읽기: `_system/`, `01_ops/strategy/`, `02_sessions/` (회고) / 쓰기: `01_ops/strategy/`, `_system/`

---

### 2.5 [클럽명]-craft (장인)

> "이것을 누구에게, 어떤 형태로 전달할 것인가?"

사고의 결과물을 **맥락에 맞는 형태로 빚는** 에이전트.
발행만이 아니라, 세션 기획안도, 내부 문서도, 멤버 가이드도 모두 "표현"이다.

- **세션 설계**: 아이디어를 구체적인 세션 기획안(진행안, 프리워크, 슬라이드 구조)으로 변환
- **콘텐츠 제작**: 공지문, 온보딩 문서, 가이드, 쇼케이스 설명, 모집 문안
- **맥락 변환**: 같은 내용을 Notion용·Discord용·대면 발표용으로 각각 변환
- **멤버 표면 설계**: 라이트 멤버가 처음 마주하는 경험을 설계

읽기: `02_sessions/backlog/`, `02_sessions/`, `05_library/research/`, `04_studio/` / 쓰기: `02_sessions/`, `01_ops/posts/`, `04_studio/`

---

### 2.6 [클럽명]-garden (정원사)

> "볼트의 토양은 건강한가?"

볼트를 **살아있는 정원처럼 돌보는** 에이전트.
파일을 정리하는 것을 넘어, 아이디어가 자랄 수 있는 환경을 가꾼다.

- **구조 관리**: 네이밍 규칙 준수 확인, 메타데이터 일관성 감사
- **연결 짓기**: 고립된 노트 발견, 관련 문서 간 `related:` 연결 제안
- **되살리기**: 아카이브에서 다시 꺼낼 가치가 있는 문서 서제스트
- **템플릿 관리**: 반복 패턴을 템플릿으로 추출, 기존 템플릿 개선
- **위생 점검**: 빈 파일, 깨진 링크, 중복 문서, 과도한 폴더 깊이 탐지

읽기: 전체 / 쓰기: 전체 (구조적 수정), `01_ops/templates/`, `04_studio/templates/`

## 3. 에이전트 간 협력 패턴

### 패턴 A: 세션 생애주기
```
thinker  →  "이 주제가 세션으로 좋겠다" (05_library/research → 02_sessions/backlog)
craft    →  세션 기획안 작성 (02_sessions)
craft    →  공지·프리워크 제작 (01_ops/posts, 04_studio)
(세션 진행)
chronicle→  세션 기록 작성 (02_sessions)
meta     →  세션 회고 패턴 분석 (01_ops/strategy)
```

### 패턴 B: 시스템 진화
```
ops      →  주간 리뷰에서 문제 발견 ("세션 포맷이 단조롭다")
meta     →  운영 R&D ("다른 커뮤니티는 어떻게 하는가?")
meta     →  개선안 제안 (01_ops/strategy)
craft    →  새 세션 포맷 기획안 (02_sessions/backlog)
garden   →  새 템플릿 생성 (01_ops/templates, 04_studio/templates)
```

### 패턴 C: 지식 축적
```
thinker  →  연구 노트 작성 (05_library/research)
garden   →  관련 기존 노트 연결 제안
chronicle→  "이번 학기에 AI 관련 연구 3편이 축적됐다" 기록
ops      →  "다음 분기는 AI 트랙을 강화하자" 제안
```

### 패턴 D: 기록과 돌봄
```
chronicle→  세션 기록 20개 축적
garden   →  아카이빙 기준에 따라 정리, 태그 보완
garden   →  "3개월 전 이 세션 인사이트가 지금 주제와 연결된다" 서제스트
thinker  →  해당 인사이트를 발전시켜 새 연구 노트 작성
```

```

  memory/
  indexes/
  logs/
```

### 출력 규칙
- 모든 출력은 해당 폴더에 마크다운 파일로 저장
- 통일된 프론트매터 양식 필수
- 에이전트 간 참조는 `related:` 필드로 연결
- 에이전트가 다른 에이전트의 작업을 요청할 때는 관련 문서에 brief를 남기거나 직접 호출
