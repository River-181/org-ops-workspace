# AX 사령탑 운영 프로토콜

> AI 사령탑(LLM 에이전트, 24/7)이 AX 총괄 사령탑으로 운영되는 환경에서 사람 운영자·AI 사령탑·Claude Code 간의 **명령 위계, 역할 경계, 승인 게이트**를 정의한다. 이 프로토콜은 [[260316-에이전트간-위임-프로토콜]](Claude Code 에이전트 간 위임)을 **보완**하며, AX 환경에서 추가되는 AI 사령탑 계층의 규칙을 담는다.

---

## 1. 명령 위계

```
사람 운영자 (감독·승인·전략)
  [운영자 목록]
      ▲
      │ 보고·승인 요청
  ┌───┴───┐
  │ AI 사령탑 │  AX 총괄 사령탑 (LLM 에이전트, 24/7)
  │ 판단·지시·통합 │
  └───┬───┘
      │ 트리거·위임
  ┌───┼───────────┐
Claude Code    외부도구·API    자가증강 스킬
(제작 스튜디오)  (Discord·Notion·Calendar)  (운영 축적)
```

| 계층 | 주체 | 역할 |
|------|------|------|
| **감독층** | 사람 운영자 | 전략 결정, 초안 승인, 자가증강 게이트 통과 여부 판단 |
| **사령탑** | AI 사령탑 (LLM 에이전트) | 클럽 운영 판단·지시·통합. 무거운 제작은 아래에 위임 |
| **실행층** | Claude Code + 외부도구 | SoT 편집, 세션 자료·설계·슬라이드 제작, API 연동 |

---

## 2. AI 사령탑의 역할 경계

### 할 수 있는 것
- Discord OPS-ROOM에서 24/7 운영 판단·지시·통합
- 세션 공지·리마인드·브리핑·기록 유도 자율 수행
- 초안·슬라이드·보고서를 생성해 운영자에게 승인 요청
- 결정사항을 `#ops-decision-log`로 라우팅
- 외부 API(Discord·Notion·Google Calendar) 직접 연동(쓰기)
- 자가증강 스킬 초안 작성 후 `#ops-system`에 검토 요청

### 할 수 없는 것 (하드 제한)
- **[볼트명](SoT) 직접 쓰기 금지** — 읽기전용 git 미러로만 참조
- 자가증강 스킬을 사람 승인 없이 자동 실행 금지
- 멤버 프라이빗 채널 메시지 수집 금지(고지된 채널 외)
- 운영자 지시 없이 멤버에게 직접 DM 발송 금지

## 3. 승인 게이트 — 사람이 반드시 개입하는 지점

| 상황 | 채널 | 운영자 행동 | 비고 |
|------|------|-------------|------|
| AI 사령탑 초안·첨부 승인 | `#ops-hermes` 또는 도메인 스레드 | ✅ → 발행 실행 / ❌ → 재작업 | 일상 승인 |
| `#announcements` 공지 발행 | `#ops-content` 스레드 | ✅ 후 발행 | Phase 2 자동화 예정 |
| 자가증강 스킬 후보 | `#ops-system` | 내용 검토 → ✅ 등록 / ❌ 재작업 | **필수 사람 게이트** |
| 볼트 편집 요청 | `#ops-command` 스레드 | 운영자 또는 Claude Code 경유 실행 | AI 사령탑 직접 쓰기 불가 |
| 새 봇 권한 확장 | ops 오프라인 결정 | 운영자 명시적 승인 + `.env` 갱신 | 최소 권한 원칙 |

## 4. 볼트 SoT 무결성 원칙

**[볼트명]는 AI 사령탑이 직접 쓰지 않는다.**

```
AI 사령탑
  → 읽기: git 미러 (git pull cron, 시크릿 미전파)
  → 쓰기 요청 시: Claude Code 또는 사람 운영자에게 지시
          ↓
    Claude Code (이 머신, SoT 읽기·쓰기)
          ↓
    볼트 파일 편집 → git commit/push
```

이 원칙을 지키는 이유:
- SoT 무결성 보호 — AI 사령탑의 실수가 볼트 원본을 훼손하지 않도록
- 시크릿 비전파 — API 키·env 값이 외부 인프라에서 볼트로 스며들지 않도록
- 감사 가능성 — 모든 볼트 변경이 사람 경유 git 이력으로 추적되도록

## 5. 자가증강 스킬 검토 루프

AI 사령탑은 문제를 해결한 뒤 재사용 가능한 스킬을 자가 작성할 수 있다(agentskills.io 표준). 이 스킬은 **초안(draft)**으로만 취급하며, 아래 루프를 거쳐야 실행 등록된다.

```
AI 사령탑 → 스킬 초안 작성
    → #ops-system 검토 큐에 게시 (멘션 @운영팀)
          ↓
    운영자 검토 (의도·범위·부작용 확인)
          ↓
    ✅ 승인 → AI 사령탑이 스킬 실행 등록
    ❌ 반려 → AI 사령탑이 재작업 후 재제출
```

**이 루프를 건너뛰는 어떤 예외도 허용하지 않는다.**

## 6. OPS-ROOM 채널 프리미티브 요약

| 채널 | 방향 | 역할 |
|------|------|------|
| `#ops-command` | 운영자 → AI 사령탑 | 자연어 지시. 과제 = 스레드 1개 |
| `#ops-hermes` | AI 사령탑 → 운영자 | 알림·초안·승인요청. 매일 모니터링 |
| `#ops-board` | AI 사령탑 브리핑 | 주간 브리핑 핀. 우선순위 보드 |
| `#ops-system` | AI 사령탑 자기점검 | 헬스·에러·자가증강 검토 큐 |
| `#ops-content` | 콘텐츠 협업 | 초안·슬라이드·첨부. 리액션으로 승인 |
| `#ops-decision-log` | 결정 기록 | append-only. AI 사령탑이 라우팅 |

## 7. 기존 에이전트 위임 프로토콜과의 관계

[[260316-에이전트간-위임-프로토콜]]은 Claude Code 에이전트 모드(ops·prep·record·research·publish) 간 파이프라인을 정의한다. AX 환경에서는 AI 사령탑이 그 위에 **사령탑 계층**으로 추가된다.

```
기존: 사람 운영자 → Claude Code 에이전트 모드 → 볼트 파일
AX:  사람 운영자 → AI 사령탑 (사령탑) → Claude Code 에이전트 모드 → 볼트 파일
                                       → 외부 API 직접 실행
```

Claude Code 에이전트 간 위임 규칙(공유 메모리, 파일 기반 핸드오프, 충돌 방지)은 그대로 유효하다. AI 사령탑은 그 위에서 조율하는 역할이지, Claude Code 내부 위임 규칙을 대체하지 않는다.

---

## 8. 멤버 응답 경계

AI 사령탑은 현재 OPS-ROOM(운영팀 전용)에서만 운영된다. 멤버 대면 채널에서의 응답은 별도 정책 문서에서 설계한다.

- 현재: 멤버 질문(`#questions`) 응대는 **운영자가 `#ops-command`에 트리거**하는 반자동 방식
- 예정: 멤버 접점 AI 사령탑 기능 정책 → [[멤버HAL-공개자료-접근정책]] (설계 예정)
- 원칙: 공개 응답은 공개 자료 기반만. 운영 내부 정보는 멤버 채널에 노출하지 않는다
