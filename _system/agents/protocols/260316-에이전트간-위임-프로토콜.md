# 에이전트 간 위임 프로토콜

## 개요

5개 에이전트 모드(ops, prep, record, research, publish)는 독립적으로 작동하되, 작업 결과가 다른 에이전트의 입력이 되는 **파이프라인 관계**가 있다.

## 핸드오프 패턴

### Pattern A: 세션 라이프사이클

```
@research → 리서치 노트 (05_library/research/)
    ↓
@prep → 세션 기획안 (02_sessions/S{NN}/plan.md)
    ↓
@publish → 세션 공지 (01_ops/posts/)
    ↓
[세션 실행]
    ↓
@record → 세션 기록 (02_sessions/S{NN}/record.md)
    ↓
@ops → 주간리뷰에서 다음 세션 준비 상태 점검
```

### Pattern B: 시스템 진화

```
@ops → 주간리뷰에서 문제 발견
    ↓
@ops → memory/active-context에 이슈 기록
    ↓
[다음 세션에서 해당 에이전트가 active-context를 읽고 대응]
```

### Pattern C: 지식 축적

```
@research → 연구 노트 (05_library/research/)
    ↓
@prep → 세션 자료에 [[wikilink]]로 연결
    ↓
@record → 세션 기록에서 실제 사용 결과 기록
    ↓
@ops → 분기 회고에서 지식 활용도 평가
```

## 위임 메커니즘

에이전트 간 직접 호출은 없다. 대신 **공유 메모리**를 통해 위임한다.

### 1. Active Context를 통한 간접 위임

`@ops`가 주간리뷰에서 다음을 기록하면:
```markdown
## 추천 작업
- @prep: S03 materials/ 채우기
- @research: PKM 방법론 비교 업데이트
- @publish: 시즌1 모집 공고 Discord 포맷팅
```

다음에 해당 에이전트가 호출될 때 active-context를 읽고 작업을 수행한다.

### 2. 파일 기반 핸드오프

에이전트가 다른 에이전트의 쓰기 폴더에 있는 파일을 읽어 상태를 파악한다.

| 생산자 | 산출물 | 소비자 |
|--------|--------|--------|
| @research | `05_library/research/*.md` | @prep (자료 링크) |
| @prep | `02_sessions/S{NN}/plan.md` | @publish (공지 소스) |
| @record | `02_sessions/S{NN}/record.md` | @ops (리뷰 입력) |
| @publish | `01_ops/posts/*.md` | @ops (발행 대기열 관리) |

### 3. 결정 로그를 통한 구조 변경 전파

구조 결정이 내려지면 `_system/agents/memory/decisions/`에 기록.
모든 에이전트는 boot 시 decisions/ 최신 항목도 확인.

## 충돌 방지

- 에이전트는 자신의 쓰기 범위 밖에 파일을 생성하지 않는다.
- 공유 영역(`_system/agents/memory/`)은 append-only. 기존 파일을 삭제하지 않는다.
- 같은 파일을 두 에이전트가 동시에 수정하는 일은 없다 (쓰기 범위가 분리됨).
