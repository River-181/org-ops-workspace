# 칸반 시스템 설계

> Obsidian Kanban 플러그인 기반 운영팀 상태 관리 시스템.
> 기존 MD 파일을 카드로 활용, frontmatter를 SoT로 유지, Google Drive 멀티유저 환경 대응.

---

## 1. 개요

### 목적

- 운영팀(ops)이 Drops, Sessions, Projects 상태를 시각적으로 관리
- 기존 MD 파일의 frontmatter `status` 계열 필드가 단일 진실 소스(SoT)
- Obsidian Bases → Notion 미러링으로 연결되는 파이프라인의 시작점

### 범위

| 포함 | 제외 |
|------|------|
| Drops / Sessions / Projects 상태 관리 | 단발성 할 일 (Tasks 플러그인이 담당) |
| ops 팀 내부 뷰 | 멤버 공개 뷰 (Notion이 담당) |
| frontmatter sync 스크립트 | 실시간 자동 동기화 |

---

## 2. 보드 구조

**3개 보드** + 1개 sync 스크립트로 구성.

| 보드 | 파일 경로 | 관리 대상 | 연결 MOC |
|------|-----------|----------|----------|
| 메인 보드 | `_system/obsidian/kanban/칸반-메인.md` | Projects, 주요 운영 이니셔티브 | `MOC-운영` |
| Sessions 보드 | `_system/obsidian/kanban/칸반-세션.md` | 세션 아이디어 → 완료 파이프라인 | `MOC-세션` |
| Drops 보드 | `_system/obsidian/kanban/칸반-드롭.md` | Drop 발행 파이프라인 | `MOC-스튜디오` |

> **네이밍 규칙 예외**: 칸반 보드 파일은 MOC, MAP, Base처럼 상시 운영 뷰 파일이므로 `YYMMDD-` 날짜 접두사를 생략한다. 날짜 접두사는 콘텐츠 문서(회의록, 세션 기록, 리서치 노트 등)에만 적용.

기존 파일과의 관계:
- `260326-드롭-운영.md` — 현행 유지 (Dataview 통계 허브). Drops 보드는 별도 파일.
- `02_sessions/backlog/` — Sessions 보드 카드의 원본 파일
- `03_projects/PRJ-*/` — 메인 보드 카드의 원본 파일 (각 프로젝트 `README.md`)

## 3. 컬럼 + Frontmatter 필드 매핑

### 3-1. Drops 보드

**필드**: `drop_status` (기존 필드, 변경 없음)

| 컬럼 표시 | frontmatter 값 |
|-----------|----------------|
| 📥 수집 중 | `inbox` |
| 🔜 발행 대기 | `ready` |
| ✅ 발행 완료 | `live` |
| ⏭ 건너뜀 | `skipped` |

### 3-2. Sessions 보드

**필드**: `session_status` (신규 추가)

| 컬럼 표시 | frontmatter 값 | 해당 파일 |
|-----------|----------------|----------|
| 💡 아이디어 | `idea` | `02_sessions/backlog/*.md` |
| 📋 기획 중 | `planning` | `S*/plan.md` (plan.md 작성 시작) |
| 🏃 준비 중 | `prep` | `S*/plan.md` (materials 작업 중) |
| ✅ 완료 | `done` | `S*/plan.md` (세션 종료 후) |

> **카드 원본 파일**: 세션 카드는 `plan.md`로 통일한다. `record.md`는 완료 후 생성되는 기록 문서이므로 칸반 카드 대상이 아님. 세션 하나당 `plan.md` 1개가 칸반 카드.

### 3-3. 메인 보드 (Projects)

**필드**: `project_status` (신규 추가)

| 컬럼 표시 | frontmatter 값 | 현재 해당 항목 |
|-----------|----------------|----------------|
| 📥 백로그 | `backlog` | PRJ-노션DB세팅, PRJ-[클럽명]-라이브러리 |
| 🔄 진행 중 | `active` | PRJ-볼트v3, PRJ-드롭시스템 |
| 👁 검토 | `review` | — |
| ✅ 완료 | `done` | — |
| ⏸ 보류 | `paused` | PRJ-워크스페이스개편 |

기존 `status: draft/live/archived` 필드는 **건드리지 않음**. 신규 필드와 공존.

## 4. 카드 형식

각 카드는 기존 MD 파일의 `[[wikilink]]`로 생성.

```markdown
- [ ] [[250926-drop-0001-huxe-ai]]
- [ ] [[S01-내우주의시작]]
- [ ] [[PRJ-볼트v3]]
```

Kanban 플러그인 카드 메타데이터로 주요 필드 표시 (읽기 전용):
- Drops: `type`, `created`, `author`
- Sessions: `created`, `author`
- Projects: `created`

## 5. 동기화 방식

### 흐름

```
ops 팀원이 Obsidian에서 카드 이동 (드래그앤드롭)
     ↓
칸반 보드 .md 파일만 변경됨 (Kanban 플러그인 기본 동작)
     ↓
ops 팀원이 sync 스크립트 실행
     ↓
_system/tools/kanban-sync.sh → kanban-sync.py
칸반 파일 파싱 → 컬럼별 [[wikilink]] 추출
     ↓
각 연결 파일의 frontmatter 업데이트
(drop_status / session_status / project_status)
     ↓
Bases 자동 반영 → (미래) Notion 미러링
```

### 실행 방법

```bash
# 워크스페이스 루트에서
_system/tools/kanban-sync.sh

# 또는 Obsidian Shell Commands 플러그인으로 단축키/버튼 연결
```

### 설계 원칙

| 원칙 | 내용 |
|------|------|
| **ops 전용** | 김주용, 김동하만 실행. 일반 멤버 실행 금지. |
| **멱등성** | 현재 값과 같으면 파일 미수정 → Google Drive 불필요한 변경 방지 |
| **단방향** | 칸반 → frontmatter 방향만. frontmatter 직접 수정도 유효. |
| **오프라인 작동** | 로컬에서 즉시 실행, 재연결 시 Drive가 동기화 |

### Google Drive 멀티유저 대응

- **fswatch/launchd 사용 안 함** — 각 멤버 머신에서 동시 실행 시 충돌 위험
- **수동 실행** — ops 팀이 명시적으로 카드 이동 후 실행
- 일반 멤버(explorers)는 칸반 파일 읽기만 가능, sync 스크립트 실행 불필요

## 6. 스크립트 설계

**위치**: `_system/tools/kanban-sync.py`

**Kanban 플러그인 MD 저장 포맷** (파싱 대상):

```markdown
---
kanban-plugin: board
---

## 📥 수집 중

- [ ] [[250926-drop-0001-huxe-ai]]
- [ ] [[260101-drop-0009-gptrecap]]

## 🔜 발행 대기

- [ ] [[260320-drop-0017-새드롭]]

%% kanban:settings
``````json
{}
```
%%
```

스크립트는 `## 컬럼명` 헤더와 `- [ ] [[wikilink]]` 패턴을 파싱한다.

```
입력: _system/obsidian/kanban/*.md (칸반 보드 파일들)
처리:
  1. 각 보드 파일 파싱 → 컬럼명과 카드([[wikilink]]) 매핑
  2. 컬럼명 → frontmatter 값 변환 (kanban-config.yaml 기반)
  3. 각 wikilink 대상 파일 찾기 (볼트 내 재귀 검색, 파일명 기준)
     - 동명 파일 충돌 시: 첫 번째 매칭 파일 사용 + 경고 로그 출력
     - 대상 파일 없음: 해당 카드 건너뜀 + 경고 로그 출력 후 계속 실행
  4. 현재 frontmatter 값 비교 → 다를 때만 업데이트
출력: 변경된 파일 목록 로그 (변경 없으면 "모두 최신 상태" 출력)
```

**설정 파일** (보드별 컬럼-값 매핑): `_system/tools/kanban-config.yaml`

```yaml
boards:
  칸반-드롭:
    field: drop_status
    columns:
      "📥 수집 중": inbox
      "🔜 발행 대기": ready
      "✅ 발행 완료": live
      "⏭ 건너뜀": skipped
  칸반-세션:
    field: session_status
    columns:
      "💡 아이디어": idea
      "📋 기획 중": planning
      "🏃 준비 중": prep
      "✅ 완료": done
  칸반-메인:
    field: project_status
    columns:
      "📥 백로그": backlog
      "🔄 진행 중": active
      "👁 검토": review
      "✅ 완료": done
      "⏸ 보류": paused
```

## 7. Bases 연계

기존 Bases를 업데이트하여 신규 필드 반영:

| Base 파일 | 추가할 필드 | 역할 |
|-----------|------------|------|
| `sessions.base` | `session_status` 컬럼 추가, 그룹화 뷰 설정 | 세션 속성 필터·정렬 |
| `projects.base` | `project_status` 컬럼 추가, 그룹화 뷰 설정 | 프로젝트 속성 필터·정렬 |
| `drops.base` (신규) | `drop_status` 기반 뷰 | Drop 속성 필터·정렬 |

> **drops.base vs Dataview 역할 분리**: `drops.base`는 drop_status·type·author 등 속성 기반 필터·정렬 뷰. `260326-드롭-운영.md`의 Dataview는 발행 통계 집계(발행/수집중/대기/보류 카운트) 전용. 기능 중복 아님.

## 8. Notion 미러링 (미래 단계)

칸반 sync 완료 후, frontmatter가 정비된 상태에서:
- `/kanban-sync` 스킬: 칸반 sync + Notion DB 업데이트 한 번에
- Drops: 기존 `/drop-publish` 스킬이 이미 Notion 발행 처리
- Sessions/Projects: 신규 Notion DB 연동 (별도 설계)
