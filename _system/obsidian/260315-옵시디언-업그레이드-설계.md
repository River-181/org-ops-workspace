# 옵시디언 업그레이드 설계

## 1. 목표

현재 워크스페이스를 Obsidian에서 다음 3개 층으로 읽히게 만든다.

1. MOC: 사람이 길을 잃지 않게 하는 허브 노트
2. Bases: 폴더 안 문서를 속성 기준으로 표처럼 읽는 코어 뷰
3. Dataview: 계산, 누락 점검, 최근 변화 추적

## 2. 표현 계층

| 대상 | 1차 표현 | 이유 | 2차 보조 |
|------|----------|------|----------|
| 조직 정체성, 운영 원칙, 시즌 구조 | MOC | 설명과 맥락이 중요 | Dataview 최근 변경 |
| 회의, 세션, 프로젝트, 재정 | Bases | 상태/담당/날짜를 정렬해야 함 | Dataview 감사 |
| 리서치 주제군, PKM 개념군 | MOC | 문맥 연결과 링크 탐색이 중요 | Bases는 선택 |
| 인박스, 초안, 누락 frontmatter | Dataview | 계산과 필터링이 핵심 | Bases triage |
| 발행물과 에셋 목록 | Bases | 발행 상태, 자산 타입 관리 | MOC 큐레이션 |

## 3. 폴더별 우선순위

| 폴더 | 우선 수단 | 설명 |
|------|----------|------|
| `_system/` | MOC + Dataview | 규칙/정체성/프로토콜 허브 + 최근 수정 감시 |
| `01_ops/` | Bases + MOC | 운영 문서는 속성 관리가 많고, 전략은 허브가 필요 |
| `02_sessions/` | Bases + MOC | 세션 카탈로그는 Base, 시즌 흐름은 MOC |
| `03_projects/` | Bases | 프로젝트 registry, owner, 상태, type이 중요 |
| `04_studio/` | Bases | 자산/exports 목록과 타입 필터링이 중요 |
| `05_library/` | MOC + Dataview | 지식 허브와 리서치 큐레이션이 핵심 |
| `06_inbox/` | Dataview + Bases | triage, 누락, 오래된 파일 확인이 중요 |

## 4. 권장 Obsidian 운영 방식

- Core plugins:
  `Properties`, `Templates`, `Backlinks`, `Bookmarks`, `Outline`, `Page preview`, `Bases`
- Community plugins:
  `Dataview` 우선
- 선택 플러그인:
  `Tasks`, `Templater`

## 5. 실행 순서

1. frontmatter 계약 통일
2. 운영/세션/프로젝트 Base 먼저 구축
3. 핵심 MOC 4-5개 배치
4. Dataview로 감사 대시보드 추가
